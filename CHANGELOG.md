# Changelog

> N=new feature, E=error fix, F=fine-tune, R=refactor, I=infrastructure, T=tag

## Unreleased
- E Android 15 / IME follow-up: MainActivity now calls `enableEdgeToEdge()`, and legacy `windowSoftInputMode="adjustResize"` was removed from the manifest. This switches the app onto the modern edge-to-edge/insets path instead of mixing enforced edge-to-edge with old resize-based IME handling
- F MemoEditor no longer auto-closes on Save. Editing an existing memo keeps the editor open and shows a green `Saved` snackbar; saving a brand-new memo now swaps `memo_new` into `memo_edit/{id}` for the created record so repeated saves update the same memo instead of creating duplicates
- R Centralize typed snackbar messages in `ScreenScaffold` via `showThemedSnackbar(...)` and `SnackbarTone` (Default/Error/Success/Info/Caution/Special). Snackbar colors are fixed globally per tone, not duplicated per app theme
- E MemoEditor white-screen, third pass — root cause: MemosHomeScreen had NO BackHandler at all, only an X button in the top bar. After Save+Back, the previous fixes correctly popped MemoEditor → Memos became visible briefly, but then the system Back (often the commit phase of a predictive-back gesture) fell through Memos into NavController, which popped the last entry off the stack. Empty NavHost → ComposeView measured 0×0 → white screen with the activity still alive. Added `BackHandler(enabled = true)` in MemosHomeScreen that routes Back through the existing onCloseRoot/onBackFromFolder callbacks (matching the top-bar close button behaviour). Also added Log.d("MemosHome") on the handler
- E MemoEditor blank-white-screen after Save+Back, second pass: BackHandler swallow alone wasn't enough — Back can still bypass it (IME consume, predictive-back gesture, dialog stacking) and pop the editor before the save coroutine completes; then onSaved() pops again → empty back stack. Added idempotent `popIfOnTop(navController, expectedRoute, refreshKey)` helper in MemListsApp.kt: pops only if the editor is still on top of the back stack, otherwise just sets the refresh flag on whichever entry is now current. Both MemoNew and MemoEdit onSaved callbacks now go through it. BackHandler swallow + canSave-stays-false from previous fix kept as first line of defense. Added Log.d("MemoEditor"/"MemListsNav") tracing on BackHandler invocation, save-coroutine completion, and the pop decision so the next white-screen repro is diagnosable from logcat
- E MemoEditor blank-white-screen after Save+Back: BackHandler was `enabled=canSave`, so during the save coroutine (canSave=false) Back fell through to NavController and popped the stack; then onSaved() popped again — empty back stack → white screen, app stuck. Now BackHandler is always enabled and swallows Back while saving; canSave stays false through the post-save navigation transition
- E Loop sound checkbox now actually limits playback to 1 cycle when unchecked. Previously the receiver always passed `repo.getSoundRepeatsSync()` regardless of `item.loopSound`, so the global setting won. Fixed in all four ReminderReceiver handlers (specific/daily/period/snoozed): repeatCount = if loopSound then settings else 1
- I Pin Gradle daemon JVM toolchain to JDK 21 via gradle/gradle-daemon-jvm.properties (generated by `updateDaemonJvm`); foojay.io download URLs included for all platforms so dev/CI use the same JVM
- E FullScreenAlertActivity no longer calls ReminderSoundService.stop in onCreate — that call is `startService(ACTION_STOP)` which the main looper processes only after onCreate returns, and its handler then calls ReminderSoundPlayer.stop(), interrupting the playback thread we just started. Now Activity only does the synchronous static cleanup `ReminderSoundPlayer.stop()` before `start()`. Promoted the static guard skip log to Log.w
- E FullScreenAlertActivity skipped sound playback when `loopSound=false && repeatCount != 1` due to a nonsensical guard `if (loopSound || repeatCount == 1)`. Dropped the guard — player always starts, ReminderSoundPlayer.start already coerces repeatCount into 1..26
- E Edited period reminder did not fire: updateMemo left `period_done_until` intact, so a previously-Done monthly period (with periodDoneUntil set to next month) would suppress the newly edited schedule via the `today < periodDoneUntil` check in ReminderScheduler.schedulePeriodItem and ReminderReceiver.handlePeriod. Fix: putNull("period_done_until") in updateMemo — same as insertMemo
- F Long-press Edit/Delete DropdownMenu on memo row shifted right via offset=DpOffset(120.dp, 0.dp) so it appears near the horizontal middle of the row instead of hugging the left edge
- F Memo home row polish: tap no longer triggers edit (swipe-right or long-press menu only, per spec 2.4); long-press opens DropdownMenu with Edit/Delete. SwipeableMemoRow wraps the card in a Box with combinedClickable(onLongClick), Card itself no longer has clickable
- F Daily/Period memos now show days-of-week in the details line before the firing time: "Mo Tu We Th Fr  09:00, 14:00" (or "Every day" when mask=127). Added formatDaysMask(daysMask, lw) in MemosFormatters; memoDetailsLine signature gains lw parameter
- F Daily time picker always opens at the current wall time, not the last added time — user expected "now" in DailyTimesEditor add-time flow
- E MemoEditorScreen Back used to need 2–3 presses to exit (IME/focus was swallowing them). Added explicit BackHandler(enabled = canSave) { onNavigateBack() } so a single press closes the editor; disabled while save is in progress to avoid racing the insert/update coroutine
- I Debug builds also produce ABI splits now (app-{arm64-v8a,armeabi-v7a,x86_64,universal}-debug.apk instead of a single app-debug.apk). 02-EmulDEBUG.sh and 03-SamsDEBUG.sh updated to glob-select the right split: emulator prefers x86_64, Samsung prefers arm64-v8a, both fall back to universal then any debug APK — same pattern as the release install scripts
- N Edit and delete memos: swipe-right on item opens editor with fields pre-populated, swipe-left shows confirmation dialog then deletes + cancels alarm. Tap on item also opens edit. MemoEditorScreen accepts optional memoId; on load reads full record via new `loadMemoForEdit(id)`, save branches to `updateMemo` vs `insertMemo` and reschedules. New route `memo_edit/{id}`, wired onEditMemo/onDeleteMemo from MemosHomeScreen. Added MemoEditable data class in MemosHomeData, new repository methods loadMemoForEdit/updateMemo/deleteMemo. SwipeToDismissBox (Material3) with confirmValueChange returning false so the row snaps back after triggering action. i18n: EditItem, Edit, Delete, Cancel, "Delete memo?"
- I Release builds split by ABI: app/build.gradle.kts gains `splits.abi { include("arm64-v8a", "armeabi-v7a", "x86_64"); isUniversalApk = true }` plus a RenameReleaseApks task that renames the four AGP outputs to `memlists-${versionName}+${versionCode}-release-{universal,arm64-v8a,armeabi-v7a,x86_64}.apk`. Task is wired via `tasks.configureEach { if (name == "assembleRelease") finalizedBy(renameReleaseApks) }`. Existing 83-RelUpload.sh already expected this naming. Install scripts updated to pick the right split: 03-EmulRELEASE prefers x86_64, 05-SamsRELEASE prefers arm64-v8a, 99-CopyToAPKX picks arm64-v8a (each with universal as fallback). 80-MakeRelease lists produced APKs after build
- F Eliminate N+1 queries on lists/memos home screens. loadListsHome previously ran one COUNT(entries) per list and loadMemoItems ran one COUNT(photos) per memo. Replaced with two private helpers loadListEntryCountsBulk(listIds) and loadPhotoCountsBulk(memoIds) that issue a single GROUP BY query keyed on the IN-clause of currently visible ids. loadListsHome collects raw rows into a private RawListRow data class first, then maps to ListContainerSummary using the bulk-fetched counts; loadMemoItems builds MemoItemSummary with photoCount=0 then copies in counts from the bulk fetch. Old per-row helpers removed
- E Battery-optimization prompt re-asks every 7 days instead of giving the user a single shot. Replaced KEY_BATTERY_DECLINED boolean with KEY_BATTERY_LAST_PROMPT timestamp + BATTERY_OPT_REPROMPT_INTERVAL_MS cooldown. shouldPromptBatteryOpt(context) returns true only when the app is still battery-optimized AND last prompt was >= 7 days ago. A user who dismissed the system screen with Back now gets another chance on the next cold start past the cooldown; once granted, isBatteryOptimized() returns false and the prompt stops. Renamed batteryOptDeclined → shouldPromptBatteryOpt and markBatteryOptDeclined → markBatteryOptPrompted in ReminderPermissions
- E Master switch (Reminders enabled) now actually cancels alarms: AppViewModel.updateRemindersEnabled awaits saveSettings then calls ReminderScheduler.cancelAll + ReminderSoundService.stop on disable, or rescheduleAll on enable. ReminderReceiver.handleDaily no longer self-reschedules when reminders are disabled — previously a daily alarm fired with master switch off would re-arm itself for the next day, leaving the app "alive" despite the toggle. Added ReminderScheduler.cancelAll(repository) helper that enumerates active items and calls cancelItem for each
- E Toggle memo `active` checkbox on memos home now syncs with AlarmManager: MemListsApp onToggleActive calls ReminderScheduler.scheduleItem on enable / cancelItem on disable, so DB and alarms no longer drift apart
- E Daily fullscreen reminder lost its `isDaily` flag in launchFullscreen overload chain — the inner overload didn't accept the parameter, so IS_DAILY extra was never put on the intent and FullScreenAlertActivity always showed the "postpone 1 day" button for daily reminders. Threaded isDaily through both overloads and added it to the intent extras; handleSnoozed call site converted to named arguments to avoid positional drift
- F Single notification during reminder playback: ReminderSoundService now reuses the reminder notification id/channel as its foreground notification, so the user no longer sees a separate "playing sound" notification appear and then disappear. On stop, the notification is re-posted as dismissable (no ongoing flag, no Stop action) and the service detaches via STOP_FOREGROUND_DETACH so it stays visible. Removed legacy "memlists_sound" channel
- F Auto-select all weekdays (mask=127) when switching reminder type to Daily or Period — empty mask was non-obvious; user can deselect days afterwards
- F Apply "Newest first" setting to Yearly/Monthly/Periods folders — previously these were hardcoded to date ASC and ignored the setting
- E Stable id-based tiebreaker in memo sort: `created` field stores only YYYYMMDD (no time), so same-day items tied and fell back to insertion order. Now the auto-increment id is the final tiebreaker — newer items appear first when "Newest first" is enabled
